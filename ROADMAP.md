# OPC 路线图

OPC 的目标不是做一个新的代码 Agent，而是把 Codex 这类执行型 Agent 放进一个可控、可恢复、可复盘的研发流程里。

第一阶段先坚持本地 CLI + 文件协议，不急着做 Web、IDE、服务端和团队化能力。

## 阶段一：最小可用闭环

目标：让一次需求可以从命令行进入 OPC，并完整走到验收和 PR 描述。

核心交付：

- `opc init` 初始化 `.opc/`。
- `opc new "需求文本"` 创建工作单并默认执行。
- 每个工作单落盘到 `.opc/work-items/<id-slug>/`。
- 工作单至少包含 `request.md`、`spec.md`、`verify.md`、`pr.md`、`state.json`。
- 编排流程固定为：需求分析 -> 实现 -> 验收 -> PR 描述。
- Agent 过程日志写入工作单目录。

完成标准：

- 一个简单需求可以自动生成规格、完成实现、产出验收报告和 PR 描述。
- 任一步产物缺失时，流程会停止并给出明确错误。
- `opc status` 能看懂当前工作单处于哪个阶段。

## 阶段二：状态机与恢复能力

目标：把流程从“能跑”提升到“失败后能恢复”。

核心交付：

- 抽出明确的状态机，集中定义状态和合法流转。
- 记录每次 Agent 运行：开始时间、结束状态、日志路径、失败原因。
- 支持 `opc resume <id>` 从中断处继续。
- 支持 Agent 失败自动重试，默认最多 2 次。
- 支持 `blocked` 状态，超过重试上限后等待人工处理。

完成标准：

- Codex 执行失败、产物无效或流程中断后，不需要重新创建工作单。
- 用户能通过 `state.json` 和日志复盘失败发生在哪一步。

## 阶段三：验收返工闭环

目标：让验收失败不再只是终止，而是能判断责任并自动返工。

核心交付：

- 规范 `verify.md` 输出格式：`status: passed` 或 `status: failed`。
- 验收失败报告必须包含失败用例、预期结果、实际结果、判断依据和建议责任方。
- 实现问题自动打回 `agent_code`。
- 需求不清、测试错误、环境问题进入明确的返工或阻塞状态。
- 支持 `opc retry <id> <agent>` 手动重跑指定阶段。

完成标准：

- 常见实现错误可以自动返工一次并重新验收。
- 无法自动归因的问题会进入 `blocked`，而不是静默失败或无限重试。

## 阶段四：测试用例独立化

目标：把“怎么验收”从实现过程中拆出来，避免测试迁就代码。

核心交付：

- 增加 `agent_test_use`，根据 `spec.md` 生成 `testcases.md`。
- `agent_code` 和 `agent_test_use` 可以在规格完成后并行执行。
- `agent_test_verify` 根据 `testcases.md` 验收实现。
- UI 任务支持保存截图到 `screenshots/`。

完成标准：

- 测试用例只依赖需求规格，不依赖实现结果。
- 验收失败时能清楚区分实现问题、测试用例问题和环境问题。

## 阶段五：项目适配与 Runner 扩展

目标：让 OPC 可以稳定接入不同项目和不同执行器。

核心交付：

- `.opc/config.json` 支持配置 `lint`、`test`、`build` 等项目命令。
- 验收阶段可以读取并执行项目命令。
- 增加 `opc doctor` 检查 runner、Git、项目命令和目录协议。
- Runner 配置支持对象形式：provider、command、args、timeout。
- 保留 Codex 默认 runner，按需扩展其他 runner。

完成标准：

- 在不同代码仓库中初始化后，只需少量配置即可使用。
- Runner 扩展不影响编排器、状态机和工作单协议。

## 暂不进入第一版

- Web 面板。
- IDE 插件。
- 服务端任务队列。
- 团队权限和审计。
- 自动创建远程 PR。
- 复杂自定义工作流引擎。

这些能力等本地闭环足够稳定后再做，否则会把核心问题掩盖在平台复杂度里。
